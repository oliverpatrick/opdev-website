01
Install Ubuntu Server 24.04 LTS
Wiping Proxmox and starting fresh on Ubuntu
Download the ISO

Get Ubuntu Server 24.04 LTS from ubuntu.com/download/server. You want the Ubuntu Server 24.04 LTS ISO (not Desktop). It's around 2.6 GB.

Create a bootable USB

Download balenaEtcher (etcher.balena.io) on your existing computer. Open it, select the Ubuntu ISO, select your USB drive (8 GB+), and flash it. This will wipe the USB.

Boot from USB and install

Plug the USB into your mini PC and boot. You may need to press a key (F2, F12, Del, or Esc depending on your hardware) to enter the boot menu. Select the USB drive.

Work through the Ubuntu installer, making these choices:

  • Language & keyboard: choose your locale
  • Network: see the dedicated step below — don't skip this
  • Proxy: leave blank
  • Storage: use the entire disk (this wipes Proxmox)
  • Profile setup: set your name, username to oli, and a strong password
  • Server name: set to greenlane
  • SSH setup: tick "Install OpenSSH server" — critical
  • Featured snaps: skip all of them

Confirm and let it install. When done, remove the USB and reboot.

Configure your static IP during the network step

When the installer reaches the Network connections screen you'll see your interface (likely enp3s0) listed with a DHCP address. You need to set it manually to 192.168.0.100 so it matches your router reservation.

  1. Select your interface (e.g. enp3s0) and press Enter
  2. Select Edit IPv4
  3. Change the method from Automatic (DHCP) to Manual
  4. Fill in the fields:
FieldValue
Subnet192.168.0.0/24
Address192.168.0.100
Gateway192.168.0.1 (your router — check the back of it if unsure)
Name servers1.1.1.1,1.0.0.1 (Cloudflare DNS — faster and private)
Search domainsleave blank

Select Save then Done. The interface should now show 192.168.0.100/24 — continue the installer.

💡
Why Cloudflare DNS? 1.1.1.1 is consistently the fastest public DNS resolver globally and doesn't log or sell your queries. Since you're already using Cloudflare for your domain and tunnel, it's a natural fit.
02
First Boot & Basic Setup
Update the system, install tools, configure the firewall
SSH in from your main machine

From your laptop or desktop, open a terminal:

bash — your local machine
ssh oli@192.168.0.100   # use your actual username and IP

Type yes to accept the fingerprint and enter your password. You're now on your server.

Update everything
bash — server
sudo apt update && sudo apt upgrade -y
Install essential tools
bash
sudo apt install -y curl wget git vim htop ufw fail2ban net-tools unzip
Add greenlane to /etc/hosts on your local machine

Run this on your laptop/desktop so you can use greenlane as a hostname instead of typing the IP every time:

bash — your local machine
echo "192.168.0.100 greenlane" | sudo tee -a /etc/hosts
ping -c 2 greenlane   # should resolve and reply

After this, ssh oli@greenlane works just like ssh oli@192.168.0.100.

Configure UFW firewall

UFW (Uncomplicated Firewall) is simple and effective. Allow SSH first so you don't lock yourself out:

bash
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 80/tcp    # HTTP (needed for Traefik ingress)
sudo ufw allow 443/tcp   # HTTPS
sudo ufw allow 6443/tcp  # Kubernetes API
sudo ufw enable          # confirm with 'y'
sudo ufw status
⚠️
Always allow SSH before enabling UFW. If you enable the firewall without allowing port 22 first, you'll be locked out and will need to access the machine directly.
⚠️
k3s and UFW conflict. k3s manages its own iptables rules for pod networking. UFW can interfere with inter-pod traffic. If you see pods that can't reach each other after enabling UFW, run sudo ufw disable to verify. For a homelab behind a NAT router, relying on the router's firewall and Tailscale ACLs is often sufficient — UFW is mainly useful for defence-in-depth if the server is directly exposed.
03
Secure SSH Access
SSH keys instead of passwords — much safer
Generate an SSH key pair on your local machine

Run this on your laptop/desktop — not the server:

bash — your local machine
ssh-keygen -t ed25519 -C "homelab"
# Press enter to accept default path (~/.ssh/id_ed25519)
# Set a passphrase (recommended) or leave empty

This creates two files: ~/.ssh/id_ed25519 (private — never share) and ~/.ssh/id_ed25519.pub (public — safe to share).

Copy your public key to the server
bash — your local machine
ssh-copy-id oli@192.168.0.100
# Enter your server password one last time

Test that key auth works before disabling passwords:

bash — local
ssh oli@192.168.0.100   # should log in WITHOUT asking for password
Harden the SSH daemon

Now that key auth works, disable password login on the server:

bash — server
sudo nano /etc/ssh/sshd_config

Find and change these lines (add them if they don't exist):

/etc/ssh/sshd_config
PasswordAuthentication no
PermitRootLogin no
PubkeyAuthentication yes
bash
sudo systemctl restart ssh
sudo systemctl status ssh   # should show Active: running

Open a new terminal window and test SSH still works before closing your current session.

Configure fail2ban

fail2ban was installed earlier. Now configure it to ban IPs that fail SSH login repeatedly:

bash — server
sudo nano /etc/fail2ban/jail.local
/etc/fail2ban/jail.local
[DEFAULT]
bantime  = 1h
findtime = 10m
maxretry = 5

[sshd]
enabled = true
port    = ssh
logpath = %(sshd_log)s
backend = %(sshd_backend)s
bash
sudo systemctl enable fail2ban
sudo systemctl start fail2ban
sudo fail2ban-client status sshd   # should show jail is active
04
Tailscale — SSH from Anywhere
VPN mesh that lets you securely reach your server from any network

Tailscale creates an encrypted mesh network between your devices. Once set up, you can SSH into your homelab from a coffee shop, on holiday, anywhere — with no port forwarding, no dynamic DNS, and no VPN complexity.

Create a Tailscale account

Go to tailscale.com and sign up (free for personal use). You'll connect devices to your "tailnet."

Install Tailscale on the server
bash — server
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up

You'll see a URL — open it in your browser to authenticate the server with your Tailscale account.

Enable Tailscale SSH

Tailscale can manage SSH authentication for you, which is even more secure:

bash
sudo tailscale up --ssh

In the Tailscale admin console (login.tailscale.com/admin), go to Machines and you'll see your server listed with a Tailscale IP (e.g. 100.x.x.x) and a MagicDNS hostname like homelab.

Install Tailscale on your laptop

Download from tailscale.com/download for your OS. Sign in with the same account. Now from anywhere:

bash — your laptop, anywhere in the world
ssh oli@greenlane          # MagicDNS hostname
# or
ssh oli@100.x.x.x       # Your Tailscale IP
💡
Tailscale is your secure backbone. Even once your services are public via Cloudflare Tunnel later, you'll keep using Tailscale for SSH. It's zero-trust: your SSH port is invisible to the public internet.

06
Install k3s — Lightweight Kubernetes
Production-grade Kubernetes that runs on a mini PC

k3s is a certified Kubernetes distribution that's packaged as a single binary. It's perfect for homelabs — it has everything you need including an ingress controller (Traefik) and storage provisioner built in.

Install k3s with a system-wide kubeconfig

k3s writes its kubeconfig to /etc/rancher/k3s/k3s.yaml. Configure it system-wide from the start so it works for any user, cron jobs, and scripts — the right approach for a server.

bash — server
# Install k3s with a system-readable kubeconfig
curl -sfL https://get.k3s.io | sh -s - --write-kubeconfig-mode=644
sudo systemctl status k3s   # should show Active: running

# Set KUBECONFIG system-wide (all users, scripts, cron)
echo 'KUBECONFIG=/etc/rancher/k3s/k3s.yaml' | sudo tee -a /etc/environment
export KUBECONFIG=/etc/rancher/k3s/k3s.yaml
💡
Why not ~/.kube/config? Copying to a home directory means only your user can run kubectl. Setting it via /etc/environment and pointing directly at the k3s file means it works system-wide — no duplication, no sync issues if the cert rotates.
Verify the cluster
bash
kubectl get nodes
# NAME       STATUS   ROLES                  AGE   VERSION
# greenlane  Ready    control-plane,master   1m    v1.32.x

kubectl get pods -A
# Lists all system pods — Traefik, CoreDNS, etc.
Install k9s — terminal UI for Kubernetes

k9s is an interactive terminal dashboard for your cluster. Browse pods, view logs, exec into containers, and delete resources — all without typing long kubectl commands.

bash — server
curl -sS https://webinstall.dev/k9s | bash
source ~/.config/envman/PATH.env
k9s   # press ? for help, q to quit
💡
k3s vs full Kubernetes: k3s is fully compatible with standard Kubernetes. Everything you learn here — kubectl, YAML manifests, Helm — transfers directly to AWS EKS, GKE, or any other cluster. It just has a smaller footprint.

Install Helm
The package manager for Kubernetes

Helm lets you install complex Kubernetes applications with a single command, using pre-built "charts" (packages). You'll use it to install Gitea, Prometheus, and Grafana.

bash
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
helm version   # version.BuildInfo{Version:"v3.x.x"...}

Add the Helm chart repositories you'll need:

bash
helm repo add gitea-charts https://dl.gitea.com/charts/
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo add argo https://argoproj.github.io/argo-helm
helm repo update
8a
Deploy Gitea — Self-hosted Git
Your own GitHub running on greenlane — pick this or 8b, not both
Create a namespace and install via Helm

Kubernetes namespaces keep your services separated and organised. Each service gets its own:

bash
kubectl create namespace gitea

helm install gitea gitea-charts/gitea \
  --namespace gitea \
  --set gitea.admin.username=admin \
  --set gitea.admin.password=ChangeThisPassword123 \
  --set gitea.admin.email=you@opdev.co.uk \
  --set ingress.enabled=true \
  --set ingress.className=traefik \
  --set 'ingress.hosts[0].host=gitea.opdev.co.uk' \
  --set 'ingress.hosts[0].paths[0].path=/' \
  --set 'ingress.hosts[0].paths[0].pathType=Prefix'
Wait for pods to be ready
bash
kubectl get pods -n gitea -w
# -w watches for changes. Wait until STATUS = Running
# Press Ctrl+C when done
Test access locally (before public DNS is set up)

Forward the Gitea port to your local machine:

bash — your local machine
ssh -L 3000:localhost:3000 oli@greenlane \
  "kubectl port-forward svc/gitea-http 3000:3000 -n gitea"

Open http://localhost:3000 in your browser. Log in with admin and your password.

⚠️
Change the admin password immediately. The first thing you should do in Gitea is set a strong password. In production, never ship with a default credential.
⚠️
Disable open registration. Once Gitea is exposed publicly via Cloudflare Tunnel, anyone on the internet can register an account by default. After logging in as admin, go to Site Administration → Users → User Management and disable registration, or add this to your Helm values: --set gitea.config.service.DISABLE_REGISTRATION=true.
8b
GitHub as your Git server
Managed Git with no server maintenance — pick this or 8a, not both

If you'd rather not maintain a Gitea instance, GitHub works perfectly with ArgoCD. You lose the fully self-hosted aspect, but gain GitHub Actions (more mature than Gitea Actions), no maintenance overhead, and a UI most developers already know.

Create a repository on GitHub

Sign up at github.com if you don't have an account. Create a new private repository called homelab-apps — this is where your Kubernetes manifests will live.

Generate a Personal Access Token

ArgoCD needs credentials to pull from your private repo:

  1. GitHub → your profile → Settings
  2. Developer settings → Personal access tokens → Tokens (classic)
  3. Click Generate new token (classic)
  4. Give it a name (e.g. greenlane-argocd), set an expiry, and tick the repo scope
  5. Copy the token — you only see it once
Push your manifests to GitHub

On greenlane, clone your new repo and push any manifests you create:

bash — server
git clone https://github.com/yourusername/homelab-apps
cd homelab-apps
mkdir -p apps
git add .
git commit -m "initial commit"
git push origin main
Connect ArgoCD to GitHub (after Phase 10)

In the ArgoCD UI go to Settings → Repositories → Connect Repo and enter:

  • Repository URL: https://github.com/yourusername/homelab-apps
  • Username: your GitHub username
  • Password: your Personal Access Token

Everything else in the guide (Mealie, ArgoCD Applications, Terraform) works identically whether you use Gitea or GitHub — just swap the repo URL.

💡
GitHub Actions vs Gitea Actions. If you choose GitHub, your CI/CD (Phase 16) uses GitHub Actions, which has a larger ecosystem of pre-built actions and free runner minutes for public repos. For private repos you get 2,000 minutes/month free.
09
Prometheus & Grafana — Monitoring
Collect metrics from your cluster and visualise them

The kube-prometheus-stack Helm chart installs everything at once: Prometheus (metrics collector), Grafana (dashboards), and AlertManager (alerts), plus automatic collection of Kubernetes cluster metrics.

Install kube-prometheus-stack
bash
kubectl create namespace monitoring

helm install monitoring prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --set grafana.adminPassword=ChangeThisPassword123 \
  --set grafana.ingress.enabled=true \
  --set grafana.ingress.ingressClassName=traefik \
  --set 'grafana.ingress.hosts[0]=grafana.opdev.co.uk'
Wait and verify
bash
kubectl get pods -n monitoring
# Takes 2-3 minutes. Wait for all pods Running.
Access Grafana locally
bash — server
kubectl port-forward svc/monitoring-grafana 3001:80 -n monitoring --address 0.0.0.0

Open http://192.168.0.100:3001 in your browser. Log in with admin and your password. Grafana comes pre-loaded with Kubernetes dashboards — explore the "Kubernetes / Compute Resources / Cluster" dashboard.

Access Prometheus directly

Prometheus has its own UI for exploring raw metrics and testing queries (PromQL). Forward it to your local machine:

bash — server
kubectl port-forward svc/monitoring-kube-prometheus-prometheus 9090:9090 -n monitoring --address 0.0.0.0

Open http://192.168.0.100:9090 in your browser. Try a query like up to see all scraped targets, or container_cpu_usage_seconds_total for CPU metrics.

💡
What you're seeing: Prometheus scrapes metrics from your Kubernetes nodes, pods, and system components every 15 seconds. Grafana reads from Prometheus and displays them. When services go public via Cloudflare Tunnel in Phase 11, you'll access Grafana via grafana.opdev.co.uk instead.
10
ArgoCD — GitOps Deployments
Deploy applications by pushing to Git — the cluster syncs automatically

ArgoCD watches a Git repository and keeps your Kubernetes cluster in sync with it. You push a YAML change to Gitea (or GitHub), ArgoCD detects it and applies it. No manual kubectl apply.

Install ArgoCD via Helm

Using the Helm chart means insecure mode and ingress are set in values — no manual patching, and upgrades are a single helm upgrade:

bash
kubectl create namespace argocd

helm install argocd argo/argo-cd \
  --namespace argocd \
  --set server.insecure=true \
  --set server.ingress.enabled=true \
  --set server.ingress.ingressClassName=traefik \
  --set 'server.ingress.hosts[0]=argocd.opdev.co.uk'
Wait for pods to be ready
bash
kubectl get pods -n argocd -w
# Wait until all pods show Running. Press Ctrl+C when done.
Get the initial admin password
bash
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d && echo

Username is admin. Change this password immediately after first login via User Info → Update Password.

Connect ArgoCD to your Git repository

In the ArgoCD UI go to Settings → Repositories → Connect Repo:

  • If using Gitea (8a): URL is https://gitea.opdev.co.uk/admin/homelab-apps — create a Gitea access token under your profile and use it as the password
  • If using GitHub (8b): URL is https://github.com/yourusername/homelab-apps — use your GitHub Personal Access Token as the password
Upgrading ArgoCD in future
bash
helm repo update
helm upgrade argocd argo/argo-cd -n argocd
11
Public Access via Cloudflare Tunnel
Expose services at opdev.co.uk with no port forwarding and automatic HTTPS

Cloudflare Tunnel runs a small daemon (cloudflared) on your server. It opens an outbound connection to Cloudflare's network. Requests to your subdomains come in via Cloudflare and flow back through that tunnel to Traefik on your server — Traefik routes to the right service. No open ports on your router required.

Add opdev.co.uk to Cloudflare

Sign up at cloudflare.com (free). Add your domain and update your domain's nameservers to point to Cloudflare (your domain registrar's control panel will have this option). This typically takes a few minutes to propagate.

Install cloudflared on your server
bash — server
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64.deb \
  -o cloudflared.deb
sudo dpkg -i cloudflared.deb
cloudflared --version
Login and create a tunnel
bash
cloudflared tunnel login
# Opens a browser to authorise — pick opdev.co.uk

cloudflared tunnel create greenlane
# Note the tunnel ID it prints — you'll need it

ls ~/.cloudflared/
# You'll see a .json credentials file with your tunnel ID as the filename
Create the tunnel config file

Replace YOUR-TUNNEL-ID with the ID from above:

~/.cloudflared/config.yaml
tunnel: YOUR-TUNNEL-ID
credentials-file: /home/oli/.cloudflared/YOUR-TUNNEL-ID.json

ingress:
  - hostname: gitea.opdev.co.uk
    service: http://127.0.0.1:80
  - hostname: grafana.opdev.co.uk
    service: http://127.0.0.1:80
  - hostname: argocd.opdev.co.uk
    service: http://127.0.0.1:80
  - service: http_status:404

All three services hit port 80 on localhost — Traefik reads the Host header and routes to the correct Kubernetes service.

Create DNS records for each subdomain
bash
cloudflared tunnel route dns greenlane gitea.opdev.co.uk
cloudflared tunnel route dns greenlane grafana.opdev.co.uk
cloudflared tunnel route dns greenlane argocd.opdev.co.uk

This creates CNAME records in your Cloudflare DNS pointing each subdomain to the tunnel.

Install cloudflared as a system service
bash
sudo cloudflared service install
sudo systemctl start cloudflared
sudo systemctl enable cloudflared
sudo systemctl status cloudflared   # should show Active: running
⚠️
Credentials move after service install. sudo cloudflared service install copies your config and credentials to /etc/cloudflared/. If you edit the config after this point, edit /etc/cloudflared/config.yaml — not the one in your home directory. The credentials file path in the config also becomes /etc/cloudflared/YOUR-TUNNEL-ID.json.
Test your services

Open these URLs in a browser — they should load with a Cloudflare-issued HTTPS certificate:

  • https://gitea.opdev.co.uk → Gitea login page
  • https://grafana.opdev.co.uk → Grafana login page
  • https://argocd.opdev.co.uk → ArgoCD login page
⚠️
Cloudflare SSL mode. In your Cloudflare dashboard under SSL/TLS, set the mode to Flexible for now (Cloudflare → your server over HTTP). Later, when you're comfortable, you can move to Full or Full (Strict) with origin certificates for end-to-end encryption.

13a
Homepage Dashboard
YAML-configured start page with live service widgets — pick this or 13b

Homepage is the most popular homelab dashboard right now. It's a clean start page that shows tiles for all your services, with live widgets that can pull stats directly from Gitea, Grafana, ArgoCD, and hundreds of other apps. You configure it via YAML files.

Create the deployment files

Create a folder on your server and write these two files:

homepage-deployment.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: homepage
---
apiVersion: v1
kind: ConfigMap
metadata:
  name: homepage-config
  namespace: homepage
data:
  services.yaml: |
    - Infrastructure:
        - Gitea:
            href: https://gitea.opdev.co.uk
            description: Git server
            icon: gitea.png
        - ArgoCD:
            href: https://argocd.opdev.co.uk
            description: GitOps deployments
            icon: argocd.png
        - Grafana:
            href: https://grafana.opdev.co.uk
            description: Monitoring dashboards
            icon: grafana.png
    - Apps:
        - Mealie:
            href: https://mealie.opdev.co.uk
            description: Recipe manager
            icon: mealie.png
  settings.yaml: |
    title: greenlane — opdev.co.uk
    theme: dark
    color: slate
  bookmarks.yaml: "[]"
  widgets.yaml: |
    - greeting:
        text_size: xl
        text: Welcome to opdev.co.uk
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: homepage
  namespace: homepage
spec:
  replicas: 1
  selector:
    matchLabels:
      app: homepage
  template:
    metadata:
      labels:
        app: homepage
    spec:
      containers:
      - name: homepage
        image: ghcr.io/gethomepage/homepage:latest
        ports:
        - containerPort: 3000
        volumeMounts:
        - name: config
          mountPath: /app/config
      volumes:
      - name: config
        configMap:
          name: homepage-config
---
apiVersion: v1
kind: Service
metadata:
  name: homepage
  namespace: homepage
spec:
  selector:
    app: homepage
  ports:
  - port: 3000
    targetPort: 3000
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: homepage
  namespace: homepage
spec:
  ingressClassName: traefik
  rules:
  - host: home.opdev.co.uk
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: homepage
            port:
              number: 3000
Deploy it
bash
kubectl apply -f homepage-deployment.yaml
kubectl get pods -n homepage -w
# Wait for STATUS = Running
Add the subdomain to Cloudflare Tunnel

Add home.opdev.co.uk to your ~/.cloudflared/config.yaml ingress block:

~/.cloudflared/config.yaml — add this line
  - hostname: home.opdev.co.uk
    service: http://127.0.0.1:80
bash
cloudflared tunnel route dns greenlane home.opdev.co.uk
sudo systemctl restart cloudflared

Open https://home.opdev.co.uk — you should see your dashboard with tiles for each service.

💡
Live widgets. Homepage can show live stats inside each tile — e.g. the number of Gitea repos, Grafana alert count, ArgoCD sync status. Add an href and a widget block per service in services.yaml. The Homepage docs (gethomepage.dev) have the exact widget config for each app.
13b

Terraform — Infrastructure as CodeOptional
Manage your Cloudflare DNS and Tailscale ACLs as versioned, repeatable code

ArgoCD already handles your Kubernetes resources via GitOps. Terraform fills the gap for everything outside the cluster — your Cloudflare DNS records and Tailscale network policies. Instead of running CLI commands or clicking dashboards, you declare what you want in .tf files, commit them to Gitea, and terraform apply makes it so.

💡
What this replaces: every cloudflared tunnel route dns greenlane *.opdev.co.uk command you've run manually. Adding a new subdomain in future becomes a one-line Git commit.
Install Terraform on your server
bash — server
wget -O- https://apt.releases.hashicorp.com/gpg | \
  sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg

echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \
  https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/hashicorp.list

sudo apt update && sudo apt install -y terraform
terraform --version
Create a terraform directory in your homelab-apps repo
bash
cd homelab-apps
mkdir terraform
cd terraform

You'll create four files here. The structure:

  • main.tf — provider config
  • variables.tf — variable declarations
  • cloudflare.tf — DNS records
  • tailscale.tf — ACL rules
  • terraform.tfvars — your actual secrets (never committed)
  • .gitignore — keeps secrets and state out of Git
Write the provider config
terraform/main.tf
terraform {
  required_providers {
    cloudflare = {
      source  = "cloudflare/cloudflare"
      version = "~> 4"
    }
    tailscale = {
      source  = "tailscale/tailscale"
      version = "~> 0.16"
    }
  }
}

provider "cloudflare" {
  api_token = var.cloudflare_api_token
}

provider "tailscale" {
  api_key = var.tailscale_api_key
  tailnet = var.tailscale_tailnet
}
Declare your variables
terraform/variables.tf
variable "cloudflare_api_token" {
  description = "Cloudflare API token with DNS edit permissions"
  sensitive   = true
}

variable "cloudflare_zone_id" {
  description = "Zone ID for opdev.co.uk (found in Cloudflare dashboard)"
}

variable "tunnel_id" {
  description = "Cloudflare Tunnel ID (from: cloudflared tunnel list)"
}

variable "tailscale_api_key" {
  description = "Tailscale API key"
  sensitive   = true
}

variable "tailscale_tailnet" {
  description = "Your tailnet name — shown in Tailscale admin under Settings > General"
}
Define your Cloudflare DNS records

This replaces every cloudflared tunnel route dns command with a single resource block. Adding a new subdomain is a one-line change.

terraform/cloudflare.tf
locals {
  subdomains = toset(["home", "gitea", "grafana", "argocd", "mealie"])
}

resource "cloudflare_record" "tunnel" {
  for_each = local.subdomains

  zone_id = var.cloudflare_zone_id
  name    = each.key
  value   = "${var.tunnel_id}.cfargotunnel.com"
  type    = "CNAME"
  proxied = true
}

To add a new subdomain in future — say uptime.opdev.co.uk — just add "uptime" to the local.subdomains set and run terraform apply.

Define your Tailscale ACL

This manages who on your tailnet can reach what. The policy below allows all members to SSH into their own devices — a sensible default to start from.

terraform/tailscale.tf
resource "tailscale_acl" "homelab" {
  acl = jsonencode({
    acls = [
      {
        action = "accept"
        src    = ["autogroup:member"]
        dst    = ["autogroup:self:*"]
      }
    ]
    ssh = [
      {
        action = "accept"
        src    = ["autogroup:member"]
        dst    = ["autogroup:self"]
        users  = ["autogroup:nonroot", "root"]
      }
    ]
  })
}
Set your credentials (not committed to Git)
terraform/terraform.tfvars

cloudflare_zone_id   = "your-zone-id"
tunnel_id            = "your-tunnel-id"
tailscale_api_key    = "tskey-api-xxxxxxxxxxxx"
tailscale_tailnet    = "youremail.gmail.com"

*.tfstate.backup

💡
From here, the workflow is: edit a .tf file → terraform plan to preview → terraform apply to ship. Adding mealie2.opdev.co.uk is one line in cloudflare.tf and a single apply. Removing it is a deletion and another apply — Terraform figures out the diff.

kubectl from Your Laptop
Manage greenlane's cluster from your local machine — no SSH required

Once your cluster is running you don't need to SSH into greenlane to run kubectl commands. Copy the kubeconfig to your laptop and manage everything remotely — including using k9s locally for a fast TUI dashboard.

Copy the kubeconfig to your laptop

Run this on your laptop/desktop. It copies greenlane's kubeconfig and rewrites the server address so your laptop can reach it:

bash — your local machine (macOS/Linux)
mkdir -p ~/.kube
scp oli@greenlane:/etc/rancher/k3s/k3s.yaml ~/.kube/greenlane.yaml

# Rewrite 127.0.0.1 → greenlane so your laptop hits the right address
sed -i '' 's/127.0.0.1/greenlane/g' ~/.kube/greenlane.yaml   # macOS
# sed -i 's/127.0.0.1/greenlane/g' ~/.kube/greenlane.yaml   # Linux

export KUBECONFIG=~/.kube/greenlane.yaml
kubectl get nodes   # should show greenlane Ready

To make this persistent across sessions, add the export to your shell profile (~/.zshrc or ~/.bashrc).

💡
You need greenlane reachable from your laptop. This works on your home network directly, or from anywhere via Tailscale (Phase 4) since Tailscale assigns greenlane a stable IP. Make sure port 6443 is open in UFW if you enabled it.
Install k9s on your laptop
bash — your local machine
# macOS
brew install derailed/k9s/k9s

# Linux / WSL
curl -sS https://webinstall.dev/k9s | bash
source ~/.config/envman/PATH.env

With KUBECONFIG pointing at greenlane, running k9s on your laptop gives you a full interactive dashboard into the remote cluster.

💡
Multiple clusters? You can merge multiple kubeconfigs using KUBECONFIG=~/.kube/greenlane.yaml:~/.kube/work.yaml and switch between them with kubectl config use-context <name>. Tools like kubectx make this even easier.

Gitea Actions — CI/CD PipelineOptional
Build and push Docker images automatically on every git push

Gitea ships with a built-in CI/CD system compatible with GitHub Actions syntax. Combined with ArgoCD, you get a full GitOps loop: push code to Gitea → Gitea Actions builds and pushes a Docker image → updates the image tag in your manifests repo → ArgoCD deploys the new version automatically.

Deploy an act_runner into your cluster

The act_runner executes your workflow jobs. Deploy it in the gitea namespace:

apps/act-runner/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: act-runner
  namespace: gitea
spec:
  replicas: 1
  selector:
    matchLabels:
      app: act-runner
  template:
    metadata:
      labels:
        app: act-runner
    spec:
      containers:
      - name: act-runner
        image: gitea/act_runner:latest
        env:
        - name: GITEA_INSTANCE_URL
          value: "https://gitea.opdev.co.uk"
        - name: GITEA_RUNNER_REGISTRATION_TOKEN
          value: "your-runner-token"
        volumeMounts:
        - name: docker-sock
          mountPath: /var/run/docker.sock
      volumes:
      - name: docker-sock
        hostPath:
          path: /var/run/docker.sock

Get the registration token from Gitea: Site Administration → Runners → Create new Runner. Replace your-runner-token above, then apply:

bash
kubectl apply -f apps/act-runner/deployment.yaml
kubectl get pods -n gitea -w   # wait for act-runner to be Running
Add a workflow to your app repo

Create .gitea/workflows/build.yaml in any repo you want to auto-build:

.gitea/workflows/build.yaml
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build and push Docker image
        run: |
          docker build -t gitea.opdev.co.uk/admin/myapp:${{ gitea.sha }} .
          docker push gitea.opdev.co.uk/admin/myapp:${{ gitea.sha }}

      - name: Update image tag in homelab-apps
        run: |
          git clone https://gitea.opdev.co.uk/admin/homelab-apps
          cd homelab-apps
          sed -i "s|image: gitea.opdev.co.uk/admin/myapp:.*|image: gitea.opdev.co.uk/admin/myapp:${{ gitea.sha }}|" apps/myapp/deployment.yaml
          git commit -am "ci: update myapp to ${{ gitea.sha }}"
          git push
🔁
The full GitOps loop: push code → Gitea Actions builds & pushes image → updates image tag in homelab-apps → ArgoCD detects the manifest change → rolls out the new version. Zero manual steps from commit to production.
💡
Using GitHub instead? Replace the workflow file path with .github/workflows/build.yaml, use github.sha instead of gitea.sha, and set up GitHub Actions — it has a larger ecosystem of pre-built actions and doesn't require a self-hosted runner for simple builds.

Good to Know
Concepts, gotchas, and commands worth having on hand

Pods, Deployments, and Services — what's the difference?

These three things are constantly referenced and easy to confuse at first:

ObjectWhat it isAnalogy
PodThe actual running container(s). Ephemeral — if it dies, it's gone.A running process
DeploymentManages pods. Ensures the right number are running, restarts them if they crash.A process supervisor
ServiceA stable network endpoint pointing at pods. Pods come and go; the Service address stays fixed.A load balancer / DNS name
IngressRoutes external HTTP traffic to a Service based on hostname or path.A reverse proxy rule
NamespaceA logical grouping — keeps services separated so names don't clash.A folder

How services find each other inside the cluster

Every Service gets a DNS name inside the cluster: service-name.namespace.svc.cluster.local. So Grafana can reach Prometheus at monitoring-prometheus.monitoring.svc.cluster.local. You rarely need to type this in full — within the same namespace, just the service name works.

Essential kubectl commands

bash
# See everything running across all namespaces
kubectl get pods -A

# Watch pods update in real time (Ctrl+C to stop)
kubectl get pods -n <namespace> -w

# See logs for a pod
kubectl logs <pod-name> -n <namespace>

# Follow logs live
kubectl logs <pod-name> -n <namespace> -f

# Detailed info including recent events (great for debugging)
kubectl describe pod <pod-name> -n <namespace>

# Open a shell inside a running pod
kubectl exec -it <pod-name> -n <namespace> -- sh

# See recent cluster events (what's been happening)
kubectl get events -n <namespace> --sort-by='.lastTimestamp'

# Check CPU and memory usage
kubectl top nodes
kubectl top pods -A

# List all services and their ports
kubectl get svc -A

What restarts automatically after a reboot

Everything critical is a systemd service and will start on boot without you doing anything:

  • k3s — starts automatically, brings all your pods back up
  • cloudflared — tunnel reconnects, subdomains come back online
  • Tailscale — reconnects to your tailnet, SSH available again
  • Docker — starts automatically

Pods managed by Deployments restart themselves if they crash. The only things that don't come back on their own are things you ran manually (like one-off kubectl port-forward commands).

UFW and k3s — a common gotcha

k3s uses iptables for pod networking. UFW can sometimes block traffic between pods on the same node. If pods can't talk to each other after you enable UFW, run:

bash
sudo ufw allow in on cni0
sudo ufw allow in on flannel.1
sudo ufw reload

Keeping everything up to date

bash
# OS — run periodically
sudo apt update && sudo apt upgrade -y

# Helm releases — update repos then upgrade each release
helm repo update
helm upgrade gitea gitea-charts/gitea -n gitea
helm upgrade monitoring prometheus-community/kube-prometheus-stack -n monitoring

# k3s — re-running the installer with the same flags upgrades in place
curl -sfL https://get.k3s.io | sh -s - --write-kubeconfig-mode=644

Security reminders

  • Never commit secrets to Gitea. API tokens, passwords, and keys belong in terraform.tfvars (gitignored) or Kubernetes Secrets. When you're ready, look into Sealed Secrets or Vault for proper secret management.
  • Disable Gitea open registration. Your Gitea is public via Cloudflare — by default anyone can sign up. Go to Gitea Admin Panel → Site Administration → Authentication Sources and disable registration, or set it to require admin approval.
  • Rotate credentials periodically. Cloudflare API tokens, Tailscale keys, and Gitea tokens should be rotated every few months.
  • Keep your OS patched. sudo apt update && sudo apt upgrade -y once a week is a good habit. Enable automatic security updates with sudo dpkg-reconfigure -plow unattended-upgrades.

YAML gotchas

  • Indentation is everything. YAML uses spaces, not tabs. Two spaces per level is the convention. One wrong space breaks the whole file.
  • Validate before applying. Use kubectl apply --dry-run=client -f file.yaml to catch errors before they hit the cluster.
  • Quotes matter for strings that look like numbers. version: 1.0 is a float; version: "1.0" is a string. Most Helm values need strings.

Quick port-forward for testing

Any time you want to access a service directly without going through Cloudflare or Traefik — for debugging, or before you've set up a subdomain — use port-forward:

bash — on greenlane
# Access a service at localhost:8080
kubectl port-forward svc/<service-name> 8080:<service-port> -n <namespace> --address 0.0.0.0

The --address 0.0.0.0 flag makes it reachable from your laptop at http://192.168.0.100:8080, not just from greenlane itself.
