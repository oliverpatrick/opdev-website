01
Install Ubuntu Server 24.04 LTS
Wiping Proxmox and starting fresh on Ubuntu
Download the ISO

Get Ubuntu Server 24.04 LTS from ubuntu.com/download/server. You want the Ubuntu Server 24.04 LTS ISO (not Desktop). It's around 2.6 GB.

Create a bootable USB

Download balenaEtcher (etcher.balena.io) on your existing computer. Open it, select the Ubuntu ISO, select your USB drive (8 GB+), and flash it. This will wipe the USB.

Boot from USB and install

Plug the USB into your mini PC and boot. You may need to press a key (F2, F12, Del, or Esc depending on your hardware) to enter the boot menu. Select the USB drive.

Work through the Ubuntu installer, making these choices:

  • Language & keyboard: choose your locale
  • Network: see the dedicated step below — don't skip this
  • Proxy: leave blank
  • Storage: use the entire disk (this wipes Proxmox)
  • Profile setup: set your name, username to oli, and a strong password
  • Server name: set to greenlane
  • SSH setup: tick "Install OpenSSH server" — critical
  • Featured snaps: skip all of them

Confirm and let it install. When done, remove the USB and reboot.

Configure your static IP during the network step

When the installer reaches the Network connections screen you'll see your interface (likely enp3s0) listed with a DHCP address. You need to set it manually to 192.168.0.100 so it matches your router reservation.

  1. Select your interface (e.g. enp3s0) and press Enter
  2. Select Edit IPv4
  3. Change the method from Automatic (DHCP) to Manual
  4. Fill in the fields:
FieldValue
Subnet192.168.0.0/24
Address192.168.0.100
Gateway192.168.0.1 (your router — check the back of it if unsure)
Name servers1.1.1.1,1.0.0.1 (Cloudflare DNS — faster and private)
Search domainsleave blank

Select Save then Done. The interface should now show 192.168.0.100/24 — continue the installer.

💡
Why Cloudflare DNS? 1.1.1.1 is consistently the fastest public DNS resolver globally and doesn't log or sell your queries. Since you're already using Cloudflare for your domain and tunnel, it's a natural fit.
02
First Boot & Basic Setup
Update the system, install tools, configure the firewall
SSH in from your main machine

From your laptop or desktop, open a terminal:

bash — your local machine
ssh oli@192.168.0.100   # use your actual username and IP

Type yes to accept the fingerprint and enter your password. You're now on your server.

Update everything
bash — server
sudo apt update && sudo apt upgrade -y
Install essential tools
bash
sudo apt install -y curl wget git vim htop ufw fail2ban net-tools unzip
Add greenlane to /etc/hosts on your local machine

Run this on your laptop/desktop so you can use greenlane as a hostname instead of typing the IP every time:

bash — your local machine
echo "192.168.0.100 greenlane" | sudo tee -a /etc/hosts
ping -c 2 greenlane   # should resolve and reply

After this, ssh oli@greenlane works just like ssh oli@192.168.0.100.

Configure UFW firewall

UFW (Uncomplicated Firewall) is simple and effective. Allow SSH first so you don't lock yourself out:

bash
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 80/tcp    # HTTP (Caddy reverse proxy)
sudo ufw allow 443/tcp   # HTTPS
sudo ufw enable          # confirm with 'y'
sudo ufw status
⚠️
Always allow SSH before enabling UFW. If you enable the firewall without allowing port 22 first, you'll be locked out and will need to access the machine directly.
03
Secure SSH Access
SSH keys instead of passwords — much safer
Generate an SSH key pair on your local machine

Run this on your laptop/desktop — not the server:

bash — your local machine
ssh-keygen -t ed25519 -C "homelab"
# Press enter to accept default path (~/.ssh/id_ed25519)
# Set a passphrase (recommended) or leave empty

This creates two files: ~/.ssh/id_ed25519 (private — never share) and ~/.ssh/id_ed25519.pub (public — safe to share).

Copy your public key to the server
bash — your local machine
ssh-copy-id oli@192.168.0.100
# Enter your server password one last time

Test that key auth works before disabling passwords:

bash — local
ssh oli@192.168.0.100   # should log in WITHOUT asking for password
Harden the SSH daemon

Now that key auth works, disable password login on the server:

bash — server
sudo nano /etc/ssh/sshd_config

Find and change these lines (add them if they don't exist):

/etc/ssh/sshd_config
PasswordAuthentication no
PermitRootLogin no
PubkeyAuthentication yes
bash
sudo systemctl restart ssh
sudo systemctl status ssh   # should show Active: running

Open a new terminal window and test SSH still works before closing your current session.

Configure fail2ban

fail2ban was installed earlier. Now configure it to ban IPs that fail SSH login repeatedly:

bash — server
sudo nano /etc/fail2ban/jail.local
/etc/fail2ban/jail.local
[DEFAULT]
bantime  = 1h
findtime = 10m
maxretry = 5

[sshd]
enabled = true
port    = ssh
logpath = %(sshd_log)s
backend = %(sshd_backend)s
bash
sudo systemctl enable fail2ban
sudo systemctl start fail2ban
sudo fail2ban-client status sshd   # should show jail is active
04
Tailscale — SSH from Anywhere
VPN mesh that lets you securely reach your server from any network

Tailscale creates an encrypted mesh network between your devices. Once set up, you can SSH into your homelab from a coffee shop, on holiday, anywhere — with no port forwarding, no dynamic DNS, and no VPN complexity.

Create a Tailscale account

Go to tailscale.com and sign up (free for personal use). You'll connect devices to your "tailnet."

Install Tailscale on the server
bash — server
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up

You'll see a URL — open it in your browser to authenticate the server with your Tailscale account.

Enable Tailscale SSH

Tailscale can manage SSH authentication for you, which is even more secure:

bash
sudo tailscale up --ssh

In the Tailscale admin console (login.tailscale.com/admin), go to Machines and you'll see your server listed with a Tailscale IP (e.g. 100.x.x.x) and a MagicDNS hostname like greenlane.

Install Tailscale on your laptop

Download from tailscale.com/download for your OS. Sign in with the same account. Now from anywhere:

bash — your laptop, anywhere in the world
ssh oli@greenlane          # MagicDNS hostname
# or
ssh oli@100.x.x.x       # Your Tailscale IP
💡
Tailscale is your secure backbone. Even once your services are public via Cloudflare Tunnel later, you'll keep using Tailscale for SSH. It's zero-trust: your SSH port is invisible to the public internet.

06
Shared Docker Network
Containers need a shared network so Caddy can route between them

In Docker, containers on the same custom network can reach each other by container name. Create one shared network that all your services will join — Caddy sits on this network and proxies requests to the right container.

Create the shared network and services directory
bash — server
docker network create web
mkdir -p ~/services
ls ~/services   # empty for now
💡
Every service in this guide joins the web network. Each docker-compose.yml declares networks: web: external: true — this attaches it to the network you just created. Caddy is also on this network, which is how it reaches containers like gitea:3000 by name.
07
Caddy — Reverse Proxy
Routes incoming requests to the right container by hostname

Caddy is the reverse proxy that sits in front of all your services. It reads the Host header on every incoming request and routes it to the correct container. Since Cloudflare Tunnel handles HTTPS externally, Caddy just needs to route HTTP on port 80 — the Caddyfile below uses http:// prefixes to disable Caddy's auto-TLS for this setup.

Create the Caddy directory and config
bash
mkdir -p ~/services/caddy
~/services/caddy/Caddyfile
{
    admin off
}

http://gitea.opdev.co.uk {
    reverse_proxy gitea:3000
}

http://grafana.opdev.co.uk {
    reverse_proxy grafana:3000
}

http://home.opdev.co.uk {
    reverse_proxy homepage:3000
}

http://mealie.opdev.co.uk {
    reverse_proxy mealie:9000
}
~/services/caddy/docker-compose.yml
services:
  caddy:
    image: caddy:latest
    restart: unless-stopped
    ports:
      - "80:80"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data
      - caddy_config:/config
    networks:
      - web

volumes:
  caddy_data:
  caddy_config:

networks:
  web:
    external: true
Start Caddy
bash
cd ~/services/caddy
docker compose up -d
docker compose logs -f   # watch for "serving initial configuration"
💡
Adding a new service is one line. Add http://newservice.opdev.co.uk { reverse_proxy newservice:PORT } to the Caddyfile, then run docker compose exec caddy caddy reload --config /etc/caddy/Caddyfile — no restart needed, Caddy picks up changes live.

Gitea — Self-hosted Git
Your own GitHub running on greenlane — pick this or 8b
Create the Gitea directory and compose file
bash
mkdir -p ~/services/gitea
~/services/gitea/docker-compose.yml
services:
  gitea:
    image: gitea/gitea:latest
    restart: unless-stopped
    environment:
      - USER_UID=1000
      - USER_GID=1000
      - GITEA__server__DOMAIN=gitea.opdev.co.uk
      - GITEA__server__ROOT_URL=https://gitea.opdev.co.uk
      - GITEA__service__DISABLE_REGISTRATION=true
    volumes:
      - gitea_data:/data
    networks:
      - web

volumes:
  gitea_data:

networks:
  web:
    external: true
Start Gitea
bash
cd ~/services/gitea
docker compose up -d
docker compose logs -f   # watch for "Starting new Web server"

Once Cloudflare Tunnel is running (Phase 10), visit https://gitea.opdev.co.uk. On first load you'll see the Gitea setup wizard — create your admin account and complete the setup.

⚠️
Registration is disabled by default. DISABLE_REGISTRATION=true is set in the compose file above, so strangers can't sign up once your Gitea is public. Create your admin account during the initial setup wizard before this takes effect.
8b
GitHub as your Git server
Managed Git with no server maintenance — pick this or 8a

If you'd rather not self-host Git, GitHub is the obvious alternative. No server to maintain, free for public and private repos, and all your services can clone from it just as easily.

Create a GitHub account and repository

Sign up at github.com if you don't have an account. Create a new repository — call it homelab or similar. This will store your service configs (docker-compose files, configs) in version control.

Generate a Personal Access Token

Go to Settings → Developer settings → Personal access tokens → Tokens (classic). Click Generate new token and give it repo scope. Copy the token — you'll only see it once.

bash — server
git config --global user.email "your@email.com"
git config --global user.name "Your Name"
# When prompted for a password during git push, use your PAT
Store your services in Git
bash
cd ~/services
git init
git remote add origin https://github.com/yourusername/homelab.git
git add .
git commit -m "initial homelab services"
git push -u origin main
💡
Version your configs, not your data. Store your docker-compose.yml and config files in Git. Never commit secrets or .env files. Docker volumes (where actual data lives) are separate from the config and aren't tracked in Git.
09
Prometheus & Grafana — Monitoring
Container and host metrics with dashboards

The monitoring stack has four parts: cAdvisor scrapes per-container metrics automatically, Node Exporter collects host-level metrics (CPU, RAM, disk), Prometheus stores everything, and Grafana visualises it.

Create monitoring directory and Prometheus config
bash
mkdir -p ~/services/monitoring
~/services/monitoring/prometheus.yml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

scrape_configs:
  - job_name: 'cadvisor'
    static_configs:
      - targets: ['cadvisor:8080']

  - job_name: 'node'
    static_configs:
      - targets: ['node-exporter:9100']

  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']
~/services/monitoring/docker-compose.yml
services:
  prometheus:
    image: prom/prometheus:latest
    restart: unless-stopped
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--storage.tsdb.retention.time=30d'
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - prometheus_data:/prometheus
    networks:
      - web

  grafana:
    image: grafana/grafana:latest
    restart: unless-stopped
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=ChangeThisPassword123
      - GF_SERVER_DOMAIN=grafana.opdev.co.uk
      - GF_SERVER_ROOT_URL=https://grafana.opdev.co.uk
    volumes:
      - grafana_data:/var/lib/grafana
    networks:
      - web

  cadvisor:
    image: gcr.io/cadvisor/cadvisor:latest
    restart: unless-stopped
    privileged: true
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:ro
      - /sys:/sys:ro
      - /var/lib/docker/:/var/lib/docker:ro
    networks:
      - web

  node-exporter:
    image: prom/node-exporter:latest
    restart: unless-stopped
    command:
      - '--path.procfs=/host/proc'
      - '--path.sysfs=/host/sys'
      - '--collector.filesystem.ignored-mount-points=^/(sys|proc|dev|host|etc)($$|/)'
    volumes:
      - /proc:/host/proc:ro
      - /sys:/host/sys:ro
      - /:/rootfs:ro
    networks:
      - web

volumes:
  prometheus_data:
  grafana_data:

networks:
  web:
    external: true
Start monitoring
bash
cd ~/services/monitoring
docker compose up -d
docker compose ps   # all four containers should show "running"
Configure Grafana data source

Once Cloudflare Tunnel is up, open https://grafana.opdev.co.uk. Log in with admin / ChangeThisPassword123 — change the password immediately. Then:

  1. Go to Connections → Data Sources → Add data source → Prometheus
  2. Set the URL to http://prometheus:9090 — the container name resolves on the shared web network
  3. Click Save & test

Import pre-built dashboards from Grafana's library by going to Dashboards → Import and entering a dashboard ID:

  • ID 193 — Docker and system overview
  • ID 1860 — Node Exporter full host metrics
💡
cAdvisor scrapes every container automatically. You don't need to configure anything per-service — cAdvisor reads from the Docker socket and exposes metrics for every running container. Once Grafana is connected to Prometheus, dashboard ID 193 shows per-container CPU, memory, and network without any extra work.
10
Public Access via Cloudflare Tunnel
Expose services at opdev.co.uk with no port forwarding and automatic HTTPS

Cloudflare Tunnel runs a small daemon (cloudflared) on your server. It opens an outbound connection to Cloudflare's network. Requests to your subdomains come in via Cloudflare and flow back through that tunnel to Caddy on your server — Caddy routes to the right container by hostname. No open ports on your router required.

Add opdev.co.uk to Cloudflare

Sign up at cloudflare.com (free). Add your domain and update your domain's nameservers to point to Cloudflare (your domain registrar's control panel will have this option). This typically takes a few minutes to propagate.

Install cloudflared on your server
bash — server
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64.deb \
  -o cloudflared.deb
sudo dpkg -i cloudflared.deb
cloudflared --version
Login and create a tunnel
bash
cloudflared tunnel login
# Opens a browser to authorise — pick opdev.co.uk

cloudflared tunnel create greenlane
# Note the tunnel ID it prints — you'll need it

ls ~/.cloudflared/
# You'll see a .json credentials file with your tunnel ID as the filename
Create the tunnel config file

Replace YOUR-TUNNEL-ID with the ID from above. All services hit port 80 on localhost — Caddy reads the Host header and routes to the correct container.

~/.cloudflared/config.yaml
tunnel: YOUR-TUNNEL-ID
credentials-file: /home/oli/.cloudflared/YOUR-TUNNEL-ID.json

ingress:
  - hostname: gitea.opdev.co.uk
    service: http://127.0.0.1:80
  - hostname: grafana.opdev.co.uk
    service: http://127.0.0.1:80
  - hostname: home.opdev.co.uk
    service: http://127.0.0.1:80
  - hostname: mealie.opdev.co.uk
    service: http://127.0.0.1:80
  - service: http_status:404
Create DNS records for each subdomain
bash
cloudflared tunnel route dns greenlane gitea.opdev.co.uk
cloudflared tunnel route dns greenlane grafana.opdev.co.uk
cloudflared tunnel route dns greenlane home.opdev.co.uk
cloudflared tunnel route dns greenlane mealie.opdev.co.uk

This creates CNAME records in your Cloudflare DNS pointing each subdomain to the tunnel.

Install cloudflared as a system service
bash
sudo cloudflared service install
sudo systemctl start cloudflared
sudo systemctl enable cloudflared
sudo systemctl status cloudflared   # should show Active: running
⚠️
Credentials move after service install. sudo cloudflared service install copies your config and credentials to /etc/cloudflared/. If you edit the config after this point, edit /etc/cloudflared/config.yaml — not the one in your home directory.
Adding a new subdomain later

The pattern for any new service is always the same three steps:

bash
# 1. Add to Caddy (then reload)
# 2. Add to /etc/cloudflared/config.yaml ingress block
# 3. Create the DNS record
cloudflared tunnel route dns greenlane newservice.opdev.co.uk
sudo systemctl restart cloudflared
⚠️
Cloudflare SSL mode. In your Cloudflare dashboard under SSL/TLS, set the mode to Flexible (Cloudflare handles HTTPS externally; your server receives HTTP). This is correct because Caddy is running HTTP-only on port 80 in this setup.

Homepage Dashboard
YAML-configured start page with live service widgets — pick this or 11b

Homepage is the most popular homelab dashboard right now. It's a clean start page showing tiles for all your services, with live widgets that can pull stats from Gitea, Grafana, and hundreds of other apps. You configure it via YAML files.

Create the Homepage directory
bash
mkdir -p ~/services/homepage/config
~/services/homepage/docker-compose.yml
services:
  homepage:
    image: ghcr.io/gethomepage/homepage:latest
    restart: unless-stopped
    volumes:
      - ./config:/app/config
    networks:
      - web

networks:
  web:
    external: true
Create the config files
~/services/homepage/config/settings.yaml
title: greenlane
headerStyle: clean
layout:
  - Infrastructure:
      style: row
  - Services:
      style: row
~/services/homepage/config/services.yaml
- Infrastructure:
    - Grafana:
        href: https://grafana.opdev.co.uk
        description: Metrics and dashboards

    - Gitea:
        href: https://gitea.opdev.co.uk

Mealie — Your First App
A worked example of adding any new service

Adding any new service to your stack follows the same four-step pattern. Mealie — a recipe manager — demonstrates the full loop from compose file to live subdomain.

Step 1 — Write the compose file
bash
mkdir -p ~/services/mealie
~/services/mealie/docker-compose.yml
services:
  mealie:
    image: ghcr.io/mealie-recipes/mealie:latest
    restart: unless-stopped
    environment:
      - PUID=1000
      - PGID=1000
      - TZ=Europe/London
    volumes:
      - mealie_data:/app/data
    networks:
      - web

volumes:
  mealie_data:

networks:
  web:
    external: true
bash
cd ~/services/mealie
docker compose up -d
Step 2 — Add to Caddyfile and reload

The Caddyfile already includes Mealie from Phase 7. If you're adding a brand new service, add a block to ~/services/caddy/Caddyfile, then reload:

bash
cd ~/services/caddy
docker compose exec caddy caddy reload --config /etc/caddy/Caddyfile
Step 3 — Register the DNS record
bash
cloudflared tunnel route dns greenlane mealie.opdev.co.uk
Step 4 — Add to cloudflared config and restart

Edit /etc/cloudflared/config.yaml and add the new ingress entry above the catchall:

/etc/cloudflared/config.yaml (add before the last line)
  - hostname: mealie.opdev.co.uk
    service: http://127.0.0.1:80
bash
sudo systemctl restart cloudflared

Visit https://mealie.opdev.co.uk. Default credentials are changeme@example.com / MyPassword — change them immediately after first login.

Good to Know
Docker commands, patterns, and self-hosting essentials

Essential Docker commands

bash
# See all running containers
docker ps

# See all containers including stopped ones
docker ps -a

# View logs for a container
docker logs <container-name>

# Follow logs live (Ctrl+C to exit)
docker logs <container-name> -f

# Restart a single service in a stack
docker compose restart <service>

# Pull latest images and recreate containers
docker compose pull && docker compose up -d

# Stop everything in a compose stack
docker compose down

# Stop and remove volumes — deletes data, be careful
docker compose down -v

# See resource usage (CPU, RAM) for all containers
docker stats

Docker Compose cheatsheet

bash
# Start services in the background
docker compose up -d

# Recreate containers after changing the compose file
docker compose up -d --force-recreate

# See status of services in the current directory's stack
docker compose ps

# Run a command inside a running container
docker exec -it <container-name> sh

# View the compose config with interpolated values
docker compose config

Where your data lives

Docker named volumes live at /var/lib/docker/volumes/ on the host. Each volume is a directory named after the volume. Back these up — they contain your Gitea repos, Mealie recipes, and Grafana dashboards.

bash
# List all volumes
docker volume ls

# See where a volume is on disk
docker volume inspect <volume-name>

# Quick backup of a volume to a tar file
docker run --rm -v <volume-name>:/data -v $(pwd):/backup \
  alpine tar czf /backup/<volume-name>-backup.tar.gz /data

Keeping everything up to date

bash
# Update a single stack manually
cd ~/services/<service>
docker compose pull
docker compose up -d

# Or run Watchtower to auto-update all containers on a schedule
docker run -d \
  --name watchtower \
  --restart unless-stopped \
  -v /var/run/docker.sock:/var/run/docker.sock \
  containrrr/watchtower --schedule "0 0 4 * * *"
⚠️
Test updates before auto-applying them. Watchtower is convenient but can break things if an image has a breaking change. For critical services like Gitea, consider pinning a specific version tag (gitea/gitea:1.22) instead of latest so you control when to upgrade.

Restart policies

All compose files in this guide use restart: unless-stopped. This means containers restart automatically after a reboot or crash — unless you explicitly ran docker compose down. Alternatives:

  • always — restarts even after you explicitly stopped it (usually not what you want)
  • on-failure — restarts only on non-zero exit code
  • no — never restarts (default if you omit the field)

Quick service access for debugging

While a service isn't yet public via Cloudflare Tunnel — or to bypass Caddy entirely — access it directly on the LAN by port-mapping to the host temporarily:

bash
# Find the container's internal port
docker inspect <container-name> | grep -A5 Ports

# Check what's exposed on the host already
ss -tlnp | grep LISTEN

Or for a quick test, add a ports entry to the compose file temporarily — e.g. "3000:3000" — then access http://192.168.0.100:3000 from your laptop.

Security reminders

  • Change default passwords immediately. Grafana ships with admin/admin, Mealie with a known default. Change these on first login.
  • Never expose Docker's socket to untrusted containers. /var/run/docker.sock gives root-level access to the host. cAdvisor and Watchtower both need it legitimately — be careful with anything else.
  • Use .env files for secrets. Put passwords and API keys in a .env file alongside docker-compose.yml and reference them as ${MY_SECRET}. Never commit .env files to Git — add it to .gitignore.
  • Keep images updated. Security patches come through image updates. docker compose pull && docker compose up -d is your friend.
  • Disable open registration on Gitea before it goes public — already set in the compose file in Phase 8a.
